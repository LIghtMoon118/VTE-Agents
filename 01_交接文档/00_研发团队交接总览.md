# 研发团队交接总览

## 1. 项目背景

VTE 智能体是“VTE 智能预警与全生命周期管理”主线下的一期专病智能体原型。它不是传统 VTE 管理系统的简单页面升级，而是希望在既有 VTE 防控、病历、护理、检验、医嘱、随访和知识库基础上，形成一个可解释、可追溯、可对接、可质控的智能增强层。

当前需要研发团队继续推进的重点，是把已经跑通的 MVP 从“本地可演示原型”推进到“可在院内测试环境联调的工程原型”。

## 3. 当前 MVP 能力

当前 MVP 已实现：

- 本地 Web 工作台。
- Node.js 后端 API。
- 单病例 VTE 风险分析。
- 规则/RAG 证据检索。
- 模型选择与 OpenAI-compatible / 本地模型适配层。
- PDF、Word、图片、文本病历导入预检和本地抽取。
- 病历审查与病历证据链分析。
- “我存在的不足 / 我面临的风险 / 我该如何应对”三栏输出。
- 路径质控：量表解释、节点提醒、质控指标、患者宣教。
- 知识图谱：基础图谱和病例激活子图。
- 院内队列和管理驾驶舱演示入口。
- Markdown / HTML 报告导出。
- 第三方系统对接字段草案。

## 4. 当前最需要研发团队接手的事项

优先级从高到低：

1. 服务化改造：把当前单文件后端拆成接口层、业务层、知识库层、模型层、审计层。
2. 病历解析服务：稳定支持 PDF、Word、图片 OCR、EMR 文本、结构化字段。
3. 结构化数据接入：落地 `patientContext` 字段映射，先从脱敏测试集和测试环境开始。
4. RAG 工程化：文献切片、向量化、混合检索、重排序、来源引用、版本管理。
5. 规则与医学逻辑审核：把演示规则拆成可配置、可版本化、可专家审核的规则库。
6. 临床验证集：建立脱敏病例验证集，覆盖 DVT、PE、围术期、肿瘤、妊娠/产褥、出血禁忌等场景。
7. 权限与审计：补充用户身份、访问权限、日志、结果留痕和人工确认回写。
8. 院内部署：形成测试环境部署脚本、服务启动方式、接口鉴权、日志路径和回滚方案。

## 5. 当前不应误解的地方

- 这不是已完成临床验证的正式产品。
- 当前风险分析主要来自本地规则、病例文本和知识库检索，不应宣传为已经具备充分大模型临床推理能力。
- 当前知识库是结构化文献矩阵和核心指南的基础接入，尚未完成正式向量数据库和全文证据追溯。
- 当前病历导入是本地预处理能力，不代表已完成院内 EMR 全接口接入。
- 当前“病历证据链”用于医疗质量与病历完整性分析，不输出法律责任判断。

## 6. 推荐交接会议议程

1. 现场运行 MVP，确认研发团队能启动。
2. 用 2-3 个脱敏病例演示：普通高危、急症线索、病历缺口明显病例。
3. 对照 `04_接口字段与数据接入说明.md` 确认一期字段。
4. 对照 `03_工程化改造任务书.md` 分配后端、前端、算法/RAG、部署、临床验证任务。
5. 明确真实病历数据不能离开院内授权环境。
6. 明确两周内第一个交付物：可在测试环境稳定运行的服务化原型 + 脱敏病例验证报告。
