# 当前功能与边界说明

## 1. 当前功能清单

### 智能体分析

- 输入病例文本或结构化 `patientContext`。
- 输出 VTE 风险分层、风险因素、出血/禁忌提示、急症线索、建议草案。
- 综合结论包含：
  - 当前病例核心问题。
  - 已知病例事实。
  - 立即优先处理。
  - 当前最需要补齐的证据。
  - 我存在的不足。
  - 我面临的风险。
  - 我该如何应对。

### 病历审查

- 对入院评估、复评、预防措施、护理记录、出院随访等流程进行核查。
- 输出已见证据、证据摘录、缺口、补强动作和说明要点。
- 用于医疗质量、病历完整性和事实证据链分析，不用于法律责任认定。

### 多模态病历导入

- 支持文本、Word、PDF、图片。
- docx 通过本地解包抽取正文。
- PDF 优先抽取文本层；无文本层时调用 macOS OCR。
- 图片调用 macOS Vision OCR。
- 识别成功后自动同步到病例输入并触发分析。

### RAG 与知识库

- `data/knowledge.json` 已接入 VTE 文献矩阵结构化条目。
- 当前检索仍是演示级关键词/RAG，不是正式向量检索服务。
- 需要升级为文献切片、向量化、混合检索、重排序和来源引用。

### 知识图谱

- 基础图谱由 VTE 文献矩阵生成。
- 病例视图采用中心辐射图，只显示当前病例激活节点和相关证据。
- 已修正“无关节点泛化点亮”的问题，但仍需临床样本验证。

### 路径质控

- 演示 Caprini、Padua、Wells DVT/PE 等量表解释。
- 展示入院、术前、术后、病情变化、出院前等提醒节点。
- 展示风险评估、出血评估、预防措施、动态复评、出院教育等质控项。

### 模型适配

- 默认本地规则/RAG。
- 可配置本地模型、Ollama 或 OpenAI-compatible API。
- 浏览器临时 API key 不写入本地配置文件。
- 真实病历不应进入外部模型。

## 2. 当前边界

当前 MVP 不能被解释为：

- 已完成临床验证的正式系统。
- 可直接替代医生判断的诊疗工具。
- 可自动开具医嘱或自动决定抗凝方案的系统。
- 已完成院内 HIS/EMR/LIS/PACS 全接口接入的产品。
- 已具备医疗器械注册所需完整质量体系的软件。

## 3. 当前技术债

| 问题 | 影响 | 下一步 |
|---|---|---|
| 后端集中在单个 `server.js` | 不利于多人开发和测试 | 拆分 route/service/rules/rag/model/audit |
| 规则为演示规则 | 临床一致性不足 | 专家审核后配置化、版本化 |
| RAG 为轻量关键词检索 | 证据召回和可追溯性有限 | 接入向量库、混合检索、重排序 |
| 文档抽取依赖本地能力 | 跨平台部署需重构 | 服务端解析/OCR 微服务 |
| 审计日志为雏形 | 不能满足院内留痕 | 接入数据库、用户身份、结果版本 |
| 前端为原型工作台 | 不能直接嵌入院内系统 | 拆出组件、定义嵌入和回写方式 |
| 模型增强不是默认能力 | 复杂病例推理仍偏弱 | 院内私有模型或脱敏后模型链路 |

## 4. 对外描述建议

稳妥表述：

> 当前为 VTE 专病智能体一期 MVP，已跑通病例分析、病历证据链、路径质控、知识图谱、RAG 和多模态病历导入的原型流程。下一步需要在院内授权环境完成数据接入、知识库工程化、权限审计和临床验证。

不建议表述：

> 已经实现自动诊断、自动治疗建议、全院上线、临床验证完成、可直接替代现有 VTE 管理系统。
