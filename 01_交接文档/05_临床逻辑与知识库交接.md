# 临床逻辑与知识库交接

## 1. 临床目标

VTE 智能体的临床价值不应停留在“给一个风险等级”。研发团队需要把输出做成临床可用的结构：

1. 当前病例为什么有风险。
2. 是否存在急症线索。
3. 是否有出血或抗凝禁忌。
4. 当前病历证据链缺什么。
5. 医生、护士、患者、管理者分别应该看到什么。
6. 需要人工确认的节点在哪里。
7. 结果如何留痕和复盘。

## 2. 当前已覆盖的临床场景

- 围术期 VTE 高危。
- 高龄、卧床、肿瘤、手术相关风险。
- D-二聚体升高。
- 急性下肢肿痛、股青肿、肢体缺血线索。
- 疑似 PE 线索：胸痛、呼吸困难、咯血、血氧下降等。
- 妊娠/产褥相关 VTE 风险。
- 出血风险和抗凝安全核查。
- 出院宣教、复评和随访。
- 病历完整性与质控缺口。

## 3. 已修正的重要问题

当前 MVP 已处理一个关键问题：非妊娠病例不得输出妊娠相关建议。后续所有规则都应遵守同一原则：只输出当前病例条件下适用的内容，不要为了“显得全面”把无关人群、无关路径和无关风险写进去。

需要重点防止：

- 无妊娠证据却输出妊娠/产科建议。
- 无活动性出血却输出高出血风险。
- 无 PE 症状却提示 PE 急症路径。
- 无肿瘤证据却点亮肿瘤相关路径。
- 无手术计划却输出围术期抗凝停启建议。

## 4. 当前知识库资产

源码中关键文件：

| 文件 | 用途 |
|---|---|
| `data/knowledge.json` | VTE 文献矩阵结构化知识库 |
| `data/knowledge.import-stats.json` | 知识库导入统计 |
| `data/rules.json` | 风险因素和出血禁忌演示规则 |
| `data/knowledge-graph.generated.json` | 文献矩阵生成的基础知识图谱 |
| `data/knowledge-graph.json` | 临床路径种子图谱 |

## 5. 知识库下一步要求

研发团队应把知识库从“JSON 演示”升级为“可审核、可追溯、可版本管理”的知识服务。

最低要求：

- 每条知识有来源。
- 每条知识有适用人群。
- 每条知识有场景标签：DVT、PE、围术期、肿瘤、妊娠、出血、出院随访等。
- 每条知识有证据等级或资料类型。
- 每条知识有版本号。
- 每条知识有审核状态。
- 输出建议时能显示引用来源。

## 6. 临床验证指标

不应只看界面是否能输出。至少需要验证：

- 风险因素抽取准确性。
- 急症线索识别准确性。
- 出血/抗凝禁忌识别准确性。
- 建议与指南和院内路径的一致性。
- 病历缺口识别准确性。
- 证据引用是否相关。
- 无关建议出现率。
- 医生采纳率。
- 护理任务可执行性。
- 预警疲劳和误报率。

## 7. 病例输出的标准骨架

建议后续所有病例分析统一保留这个骨架：

1. 当前病例核心问题。
2. 已知病例事实。
3. 主要风险因素。
4. 急症线索。
5. 出血/抗凝安全核查。
6. 当前证据链缺口。
7. 我存在的不足。
8. 我面临的风险。
9. 我该如何应对。
10. 需要医生确认的事项。
11. 证据来源。
12. 审计编号。

这比单纯输出“高危/中危/低危”更接近真实临床使用。
