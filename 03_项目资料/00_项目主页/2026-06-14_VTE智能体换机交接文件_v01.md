## 一、项目一句话

本项目当前优先建设“VTE 专病智能体”，不是先做传统 VTE 管理系统。智能体定位为可演示、可部署、可对接的智能增强层，可接入已有 VTE 管理系统、EMR、护理系统或未来自研 VTE 管理系统。

当前一期目标：

- 企业对接前有真实可运行原型，而不是 PPT。
- 知识库先接入既有 VTE 文献矩阵。
- 保留内网部署和数十万份病历批量分析能力。
- 模型层不绑定 GPT 或某一个模型，而是可自动识别本地模型，也可配置外部或内网 OpenAI-compatible 模型。

## 五、模型层当前设计

当前智能体不绑定某一个大模型。

默认配置：

```text
MODEL_PROVIDER=auto
```

含义：

- 自动探测本机是否已有模型服务。
- 不自动调用外部网络模型。
- 没有本地模型、没有网络账号时，规则分析、RAG 检索和病历审查仍可运行。

已支持自动探测：

- Ollama: `http://localhost:11434`
- LM Studio: `http://localhost:1234/v1`
- vLLM: `http://localhost:8000/v1`
- Xinference: `http://localhost:9997/v1`
- LocalAI: `http://localhost:8080/v1`

外部或内网模型配置示例：

```bash
MODEL_PROVIDER=openai_compatible
MODEL_BASE_URL=https://api.openai.com/v1
MODEL_NAME=填写模型名
OPENAI_API_KEY=填写密钥
npm start
```

院内私有化模型示例：

```bash
MODEL_PROVIDER=openai_compatible
MODEL_BASE_URL=http://内网模型服务地址/v1
MODEL_NAME=内网模型名
MODEL_REQUIRES_API_KEY=false
npm start
```

重要边界：

- 真实病历默认禁止进入外部模型。
- 外部模型仅可用于模拟病例、脱敏病例和公开文献。
- 真实患者数据应在院内服务器、院内模型或医院授权环境处理。

## 七、和企业对接时的推荐说法

可以这样说：

> 当前演示的是 VTE 智能体一期 MVP，不是传统 VTE 管理系统。它已经具备后端 API、知识库、规则库、病历审查、RAG 问答、模型适配层和第三方系统对接能力。未来它可以接入医院已有 VTE 管理系统，也可以接入我们二期建设的 VTE 管理系统。智能体本身不绑定 GPT 或某个国产模型，而是通过模型适配层按医院部署环境选择本地模型、私有化模型或外部授权模型。

不建议说：

- 已经完成真实病历训练。
- 已经能替代医生判断。
- 已经形成临床诊疗结论。
- 已经完成医疗器械注册级验证。

## 十、换电脑后找不到资料时的排查

如果 MVP 启动失败：

- 先确认 Node.js 是否安装：`node -v`
- 进入 `vte-agent-mvp` 目录再运行 `npm start`
- 如果端口占用，使用 `PORT=8788 npm start`
- 如果知识库为空，检查 `data/knowledge.json`
- 如果需要重新导入文献矩阵，运行：

```bash
npm run import:vte-matrix
```

如果找不到 VTE 文献矩阵：

- 检查 iCloud 是否同步完成。
- 检查路径是否存在：

```text
/Users/zgj/Library/Mobile Documents/com~apple~CloudDocs/Research/科研/VTE文献矩阵
```

如果换电脑后用户名不是 `zgj`：

- 所有绝对路径中的 `/Users/zgj/` 需要替换为新电脑用户目录。
- 代码运行本身不依赖 `zgj` 用户名，但文献矩阵导入脚本可能需要同步调整来源路径。
