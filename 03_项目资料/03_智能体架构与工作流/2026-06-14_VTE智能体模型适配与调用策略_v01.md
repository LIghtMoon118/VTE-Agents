# VTE 智能体模型适配与调用策略

建立日期：2026-06-14  
版本：v01

## 一、核心判断

VTE 智能体应设计为“可直接调用模型，但不强制调用模型”的架构。智能体本体包括知识库、规则库、病历解析、流程核查、接口规范和审计日志；大模型作为增强层接入，用于复杂文本理解、生成表达、教学解释、病历摘要和多轮交互。

因此，智能体不应绑定 GPT，也不应绑定某一个本地或国产模型，而应采用模型适配层：

- 使用者机器上已有本地模型时，自动识别并优先对接。
- 无本地模型但有网络和账号时，可配置外部 OpenAI-compatible 模型。
- 无网络、无账号或医院未授权时，规则分析、RAG 检索和流程审查仍可运行。
- 真实病历默认不进入外部模型；院内服务器阶段应优先对接本地或私有化模型。

## 二、推荐产品表达

对企业或医院信息部门可以这样表述：

> VTE 智能体采用模型无关的适配器架构。系统既可以在没有大模型的情况下完成规则化风险识别、知识库检索和流程质控，也可以在授权环境下调用本地模型、私有化模型或外部 OpenAI-compatible 模型。模型是否调用、调用哪一个模型，取决于医院部署环境、数据安全要求和账号权限，而不是由智能体产品本身锁死。

## 三、模型层分工

| 模块 | 不调用大模型时 | 调用大模型后 |
| --- | --- | --- |
| 风险识别 | 规则库和结构化变量识别 | 补充非结构化病程、护理记录和出院小结理解 |
| RAG 知识库 | 关键词/向量检索返回证据 | 结合证据生成更自然的解释和问答 |
| 病历审查 | 按流程节点检查缺失项 | 生成更完整的病历质量改进建议 |
| 教学培训 | 输出固定教学要点 | 针对病例生成交互式提问和讲解 |
| 对接系统 | REST API 输出结构化结果 | 在结构化结果基础上生成摘要和说明 |

## 四、部署策略

### 1. 企业演示阶段

使用模拟病例、脱敏病例和公开文献。可用外部模型增强表达，但需明确不输入真实患者隐私数据。

### 2. 医院内网试运行阶段

在内网服务器部署智能体，接入院内数据源。模型优先采用院内本地模型或私有化模型，所有病例数据留在院内。

### 3. 商业化推广阶段

把模型层作为标准适配器交付。医院可根据本院政策选择模型服务，VTE 智能体保持知识库、流程规则、接口规范、审计日志和临床边界稳定。

## 五、一期 MVP 已落实内容

当前演示原型已完成模型适配层雏形：

- 默认 `MODEL_PROVIDER=auto`。
- 自动探测 Ollama、LM Studio、vLLM、Xinference、LocalAI 等本地模型服务。
- 支持外部或内网 OpenAI-compatible 接口。
- 提供 `GET /api/model/status` 查看模型状态。
- 提供 `POST /api/model/complete` 手动触发模型调用。
- 默认拦截 `containsRealPatientData=true` 的模型调用测试。

## 六、下一步

- 增加模型调用审计日志，记录调用时间、模型名、输入类型、输出边界和调用人。
- 增加模型输出结构化校验，避免模型直接生成医嘱或越过医生确认。
- 增加院内模型适配清单，根据医院信息中心允许的模型服务配置。
- 将模型调用与 RAG 证据链绑定，要求重要输出能追溯到指南、共识、论文或院内路径。
