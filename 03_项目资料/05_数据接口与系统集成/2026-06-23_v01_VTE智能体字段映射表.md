# VTE 智能体字段映射表 v01

建立日期：2026-06-23

用途：用于院内信息中心、企业技术团队、合作研发方逐项核对数据接入与结果回写。当前版本强调“一期可联调”，不是一次性穷尽全院所有字段。

## 一、使用原则

1. 一期先接高价值字段，不追求全量。
2. 先满足单病例分析、病历审查、路径质控和基础回写，再扩展驾驶舱和批量分析。
3. 能结构化的字段优先结构化；文本字段只作为补充，不作为长期唯一输入。
4. 所有患者标识默认使用脱敏标识，不直接用住院号、身份证号、手机号等可识别信息。

## 二、一期最小可用字段集

如果只做一期最小闭环，建议先确认以下字段能稳定获取：

- `patientToken`
- `encounterToken`
- `department`
- `admissionTime`
- `age`
- `sex`
- `diagnoses`
- `procedures`
- `D-二聚体`
- `血小板`
- `Hb`
- `PT/APTT`
- `肌酐`
- `orders`
- `nursingRecords`
- `freeText`

## 四、推荐标准化规则

### 1. 诊断

- 优先保留中文临床诊断名称。
- 如果已有 ICD 编码，可附带在原系统侧保存，但一期对接可先传名称。
- 建议区分：
  - 当前主要诊断
  - VTE 相关诊断
  - 肿瘤/感染/创伤/妊娠等风险背景诊断

### 2. 手术/操作

- 手术名称尽量保持临床原意，不要过早压缩成泛化标签。
- 可在中间层增加标准化标签，如：
  - `腹盆腔肿瘤手术`
  - `血管介入`
  - `骨科大手术`

### 3. 检验

- 建议在中间层统一检验名称。
- 建议至少保留：
  - 检验名称
  - 数值或“升高/降低”
  - 单位
  - 检验时间

### 4. 护理记录

- 一期不要求全结构化，可先用摘要。
- 但建议在中间层尽量能拆出：
  - 卧床/活动状态
  - 机械预防执行
  - 宣教完成情况
  - 复评是否完成

## 六、触发节点与字段需求对应表

| 触发节点 | 最低输入字段 | 推荐输出 |
|---|---|---|
| 入院24小时内 | `patientToken`、`encounterToken`、`department`、`age`、`sex`、`diagnoses`、`labs`、`freeText` | `riskLevel`、`riskFactors`、`bleedingFlags`、`reviewItems` |
| 术前 | 上述字段 + `procedures` + `orders` | `riskPrediction`、`recommendations`、`workflowReminders` |
| 术后24-72小时 | 上述字段 + `nursingRecords` | `roleTasks.nurse`、`workflowReminders`、`qualityIndicators` |
| 病情变化/急症触发 | `freeText` 或急症相关结构化记录 | `urgentFlags`、急症优先建议、影像/会诊提示 |
| 出院前 | `diagnoses`、`orders`、`nursingRecords`、`freeText` | `patientEducation`、`roleTasks.patient`、延长期预防复核项 |

## 七、建议的中间层转换

如果原始系统字段命名复杂或多厂商不一致，建议先做一层中间标准化，而不是让智能体直接读各系统原始字段。

推荐中间层对象：

```json
{
  "patientContext": {
    "patientToken": "...",
    "encounterToken": "...",
    "department": "...",
    "admissionTime": "...",
    "age": 0,
    "sex": "...",
    "diagnoses": [],
    "procedures": [],
    "labs": [],
    "orders": [],
    "nursingRecords": [],
    "freeText": "..."
  }
}
```

这样做的好处：

- 便于适配不同医院和不同厂商。
- 便于后续加字段而不改智能体主接口。
- 便于做审计、脱敏和版本控制。

## 八、当前不建议一期强行纳入的字段

以下字段有价值，但不建议一期就强行做全：

- 全量费用数据。
- 全量影像原始 DICOM。
- 全量护理细颗粒打点记录。
- 全量随访事件流。
- 全量消息推送轨迹。

原因：

- 联调成本高。
- 对一期临床价值提升不一定成比例。
- 会拖慢验证速度。

## 九、联调时需要逐项确认的问题

1. 哪些字段已有稳定接口，哪些只能导出表。
2. 检验名称是否需要做院内字典映射。
3. 护理系统能否提供摘要字段，还是只能给原始记录。
4. 医生确认后的结果回写到哪里。
5. 审计日志由哪一侧保存。
6. 是否允许在测试环境保留脱敏文本。

## 十、建议的下一步产物

如果继续往前推进，最应该补的是两份表：

1. 华西实际字段对照版。
2. 企业联调任务分工表。
