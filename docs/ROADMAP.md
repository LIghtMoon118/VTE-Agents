# VTE 智能体 MVP 升级路线

## 一、当前版本

当前版本是一期 MVP，具备：

- 本地 Web 工作台。
- Node.js 后端 API。
- 已接入 VTE 文献矩阵结构化知识库。
- 演示规则库。
- 模拟病例。
- 单病例分析。
- RAG 问答。
- 电子病历深度审查。
- PDF、Word、图片和文本病历导入预检入口。
- 院内队列筛选、批量分析和管理者驾驶舱演示入口。
- 对接字段草案。
- 审计日志雏形。

## 二、下一步优先级

### 1. 深化真实 VTE 文献知识库

当前已从 VTE 文献矩阵导入结构化条目。下一步将用户已有近 20 年 VTE 相关文献、指南共识、SCI 和中文核心期刊论文进一步整理为：

- 文献元数据。
- 知识条目。
- 证据等级。
- 适用场景。
- 适用人群。
- 推荐内容。
- 禁忌和注意事项。
- 审核状态。

MVP 可先用 JSON 或 SQLite，后续升级为向量数据库。

### 2. 建立正式 RAG 检索层

当前为关键词检索演示。下一步应升级为：

- 文献切分。
- 向量化。
- 混合检索。
- 重排序。
- 来源引用。
- 版本管理。
- 幻觉控制。
- 人工审核。

### 3. 扩展病例分析能力

当前支持文本或结构化摘要。下一步应支持：

- PDF 病历解析。
- Word 病历解析。
- 扫描图片/OCR。
- EMR 文本。
- 诊断编码。
- 医嘱。
- LIS。
- 护理记录。
- 手术麻醉记录。
- 超声/CTPA 报告摘要。
- 病历时间轴。
- 医疗质量和病历完整性审查。

病例输出策略应避免泛化：只点亮当前病例直接相关节点。年轻妊娠患者不应出现高龄节点；无活动性出血、血小板正常、肝肾功能可时，不应点亮出血/禁忌节点；无呼吸困难、胸痛、咯血等线索时，不应点亮 PE 急症节点。

### 4. 对接第三方 VTE 管理系统

优先定义：

- 输入字段。
- 输出字段。
- 调用频率。
- 鉴权方式。
- 日志审计。
- 错误码。
- 人工确认回写机制。

### 5. 内网服务器部署

内网部署应增加：

- Linux 服务化启动。
- 任务队列。
- 数据库。
- 权限控制。
- 访问日志。
- 结果导出分级。
- 断点续跑。
- 批量病例分析。
- 文件解析/OCR 服务。
- 院内系统接口任务编排。
- 管理者驾驶舱数据库和指标口径。

### 6. 临床验证

至少评价：

- 风险因素抽取准确性。
- 出血禁忌识别准确性。
- 建议与指南/院内路径一致性。
- 医护采纳率。
- 预警疲劳。
- 漏评漏防识别能力。
- 真实世界结局和安全事件。

### 7. 知识图谱与 AI-native 功能扩展

下一阶段不应只加强问答能力，还应扩展：

- VTE 知识图谱。
- 病例子图和相关性分层显示。
- 病历深度分析。
- 诊疗规范审查。
- 法律/医疗质量风险提示。
- 教学训练和考核。
- 科研队列构建。
- 批量病例质控复盘。
- 管理者驾驶舱：科室分布、高危患者、急症线索、漏评漏防、护理执行、出院随访和科研变量缺失。

## 三、企业合作任务拆分

临床团队负责：

- 临床问题定义。
- 知识库审核。
- 变量体系。
- 结局定义。
- 验证指标。
- 合规边界。

企业团队负责：

- 私有化部署。
- 工程架构。
- 接口适配。
- RAG 工具链。
- 工作流编排。
- 权限与审计。
- 性能优化。
- 运维升级。

## 四、不可妥协边界

- 不以聊天机器人替代 VTE 智能体。
- 不把真实病历带出院。
- 不自动开立医嘱。
- 不将智能体绑定在单一厂商系统中失去可迁移性。
- 不把演示效果等同于临床验证结果。
